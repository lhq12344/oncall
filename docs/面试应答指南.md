## Q1：Eino 框架不是自带了 Checkpoint 存储吗，为什么你还要用 Redis 来保存？

### 推荐短答

Eino 提供的是 `Checkpoint` 能力和 `CheckPointStore` 接口，不是强绑定某一种持久化介质。我的项目里用 Redis 做 `CheckPointStore` 的具体实现，是为了满足跨请求恢复、进程重启恢复和多实例共享状态这些生产场景要求。

### 展开回答

- 在后端 Controller 初始化 Runner 时，我把 Eino 的 `CheckPointStore` 接口接到了自定义的 Redis 实现上：
  - `internal/controller/chat/chat_v1.go`
  - `internal/context/checkpoint_store.go`
- Redis 存储的价值主要有三点：
  - **跨请求恢复**：中断发生后，用户确认通常是另一个 HTTP 请求，不能只依赖当前进程内存。
  - **服务重启可恢复**：即使后端重启，只要 Redis 里的 checkpoint 还在，就能继续。
  - **多实例可共享**：如果后端将来扩成多副本，checkpoint 也能被不同实例读取。
- 所以这里不是重复造轮子，而是把 Eino 提供的抽象落到了适合线上场景的存储介质上。

### 代码落点

- `internal/controller/chat/chat_v1.go`
- `internal/context/checkpoint_store.go`

## Q2：中断发生时，Eino 框架都保存了哪些信息来保证中断可完整恢复？

### 推荐短答

它保存的不是单一“中断标记”，而是整份执行快照，包括图执行状态、节点输入、子图状态，以及中断点地址和中断状态；我这边再配合工具内部状态序列化，保证恢复后不会丢失执行进度和校验上下文。

### 展开回答

- Eino 内部 checkpoint 结构中，核心会保存这些信息：
  - `Channels`：图里各节点之间的数据通道状态
  - `Inputs`：节点输入
  - `State`：图状态
  - `SkipPreHandler` / `RerunNodes`：恢复执行时的控制信息
  - `SubGraphs`：子图 checkpoint
  - `InterruptID2Addr` / `InterruptID2State`：中断 ID 到执行地址、局部状态的映射
- Eino 的中断状态本身还包含：
  - `Info`：给用户展示的中断信息
  - `State`：组件局部状态
  - `LayerSpecificPayload`：框架层附带信息
- 我在项目里做的补强有两点：
  - 恢复时通过 `ResumeWithParams(...)` 把 `approved / resolved / comment` 精确投递到目标中断点；
  - 对 execution / rca 等工具内部状态做可序列化保存，避免 checkpoint 恢复后重试计数、缓存结果丢失。

## Q3：如何避免各 Agent 之间的状态污染？

### 推荐短答

核心做法是把“用户对话历史”和“故障处置状态”物理隔离：对话历史走 `SessionMemory`，工作流内部状态走 `IncidentState`，两者分开存、分开裁剪、分开回灌。

### 展开回答

- 对话链路用 `SessionMemory`，只保存用户和助手可见的会话轮次。
- 故障工作流内部用 `IncidentState`，保存观测摘要、RCA 结论、执行状态、重试计数、最终报告等结构化数据。
- 在 `ops` 工作流里，我没有把全量 agent 历史再次回灌给模型，而是通过 `incidentHistoryRewriter` 只保留：
  - 最新用户输入
  - 当前 `IncidentState` 的结构化快照
- 这样做的结果是：
  - 大日志、大对象不会污染普通对话上下文；
  - 多 Agent 之间的状态传递依赖结构化字段，不依赖自然语言隐式约定；
  - 故障处置状态和聊天状态不会互相串味。

### 代码落点

- `internal/context/session_memory.go`
- `internal/agent/ops/state_bridge.go`
- `internal/agent/ops/incident_workflow.go`

## Q4：怎么实现摘要压缩的，怎么保证不会丢失重要信息？

### 推荐短答

我做的不是单层摘要，而是“两层压缩”：先做 token 预算硬裁剪，再做会话级摘要；另外关键故障状态不依赖摘要，而是单独存在 `IncidentState` 里，所以重要信息不会只押在摘要上。

### 展开回答

- **第一层：请求前 token 预算裁剪**
  - 在 `mem.GetMessagesForRequest(...)` 里先算 `turnsBudget`
  - 然后用 Redis + Lua 按 `Turn` 粒度从最旧历史开始裁剪
  - 这是硬控制，目的是保证请求一定不超窗
- **第二层：会话级摘要压缩**
  - `SessionMemory.compactMessages(...)` 会把更早的历史轮次压成一条“历史会话摘要”
  - 当前策略是：
    - `system` 消息永远保留
    - 最近 `20` 轮原样保留
    - 超过 `40` 轮后，把更早轮次做摘要
  - 这里主路径里的摘要不是重新调一个 LLM，而是规则式摘要：按顺序拼接 `- 用户:` / `- 助手:` 内容
- **怎么减少丢信息风险**
  - 不是按 message 裁，而是按 `Turn` 裁，尽量保留问答完整性；
  - 近期上下文高保真，远期上下文才摘要；
  - 关键故障状态、执行结果、RCA 结论不依赖聊天摘要，而是走 `IncidentState`；
  - 摘要只是降低上下文成本，不承担唯一事实来源。

### 面试时要注意的表述

- 不建议说“绝对不会丢信息”；
- 更准确的说法是：**关键结构化状态不依赖摘要，摘要只作用于远期对话历史，因此重要信息的丢失风险被隔离了。**

### 代码落点

## Q5：你说可以过滤 LLM 的 reasoning / thinking，怎么实现过滤的？

### 推荐短答

我在消息进入历史前统一做净化，默认把 `ReasoningContent` 清空，不让 thinking block 进入后续上下文。

### 展开回答

- `mem` 默认配置里 `KeepReasoningInContext=false`
- 每条消息在进入上下文前，都会经过 `sanitizeForContext(...)`
- 这个函数会做三件事：
  - 清理 `ResponseMeta`
  - 清理 `Extra`
  - 如果 `KeepReasoningInContext=false`，就把 `ReasoningContent=""`
- 这样做的原因是 reasoning 往往很长，token 成本高，而且它对后续轮次未必比结构化结论更稳定。

### 代码落点

- `utility/mem/mem.go`

## Q6：多 Agent 有保存历史吗，Agent 的输入怎么保证不超过窗口大小？

### 推荐短答

有历史，但不是把所有 Agent 历史无脑回灌。不同链路用不同策略：对话链路保留对话历史并做预算裁剪，运维工作流链路通过 `HistoryRewriter` 只保留必要输入和结构化状态。

### 展开回答

- **对话链路**
  - 会把用户问答 turn 存到 Redis
  - 下次请求前先算 token 预算，再裁剪历史，再做会话层 compact
- **多 Agent 工作流链路**
  - ADK 自身会积累 `HistoryEntry`
  - 但项目没有把整条 agent 历史完整回灌给模型
  - 我通过 `incidentHistoryRewriter` 只保留：
    - 最后一条用户输入
    - 当前结构化的 `IncidentState`
- **如何保证不超窗**
  - 对话链路：`turnsBudget = MaxInputTokens - ReserveOutputTokens - ReserveToolsTokens - UserTokens - SafetyTokens - SysTokens`
  - 历史按 `Turn` 原子裁剪
  - 关键长文本通过 `clipText(...)`、`latestIncidentLogs(...)` 等方式裁剪
  - token 计算优先调用精确分词接口，失败时再本地估算

- `internal/context/session_memory.go`
- `utility/mem/mem.go`

## Q7：受控命令执行引擎中静态风险分析的正则匹配是怎么做的？

### 推荐短答

我把计划里的命令分成三类：绝对禁止、高风险需审批、只读安全；实现上是把每个执行步骤渲染成完整命令文本后，用三组正则规则做分类匹配。

### 展开回答

- `validate_plan` 会先把步骤渲染成完整命令字符串
- 然后依次匹配三组规则：

#### 1）绝对禁止

- `rm -rf /`
- `mkfs`
- `dd if=`
- `shutdown / reboot`
- fork bomb

#### 2）高风险变更

- `kubectl delete / drain / cordon / uncordon / scale / rollout restart / patch`
- `docker stop / restart / rm`
- `systemctl stop / restart / disable`
- `helm upgrade / rollback / uninstall`

#### 3）只读操作

- `kubectl get / describe / logs / top`
- `curl / wget`
- `cat / ls / ps / ss / netstat / top / df / du / free / uptime / echo`

- 如果命中禁止规则，返回 `blocked=true`
- 如果命中高风险规则，返回风险原因，后续执行阶段逐步触发审批
- 如果都不命中，再结合其它条件计算 `risk_level`

### 面试时的表达建议

- 不要说“靠 AI 识别危险命令”
- 要说：**静态风险分析是后端规则引擎做的确定性判断，LLM 不参与最终风险裁决**

### 代码落点

- `internal/agent/execution/tools/validate_plan.go`

## Q8：怎么知道执行的命令是高风险的，LLM 判断吗？

### 推荐短答

不是。LLM 只负责提出候选动作，真正的风险判断由后端规则和执行引擎决定。

### 展开回答

- **第一层：计划阶段判断**
  - `validate_plan` 通过正则和规则把命令分成禁止、高风险、只读三类
- **第二层：执行阶段再次判断**
  - `execute_step` 会根据具体命令和参数判断当前步骤是否需要审批
  - 例如：
    - `kubectl` 是否只读，取决于 verb 是不是 `get / describe / logs / top`
    - `docker` 是否只读，取决于是不是 `ps / images / logs / inspect / stats`
    - `systemctl` 是否只读，取决于是不是 `status / show / list-units`
    - `bash` 脚本会按 `|| && ; | \n` 拆段后逐段判断是否只读
- 只有规则判断为“可能修改资源或运行状态”的命令，才会触发审批中断

### 代码落点

- `internal/agent/execution/tools/validate_plan.go`
- `internal/agent/execution/tools/execute_step.go`

### 面试时可以这样总结

> 我把 LLM 放在“规划层”，把风险判断放在“规则层”，把最终执行放在“审批层”。  
> 也就是说，模型可以建议执行什么，但不能决定这条命令是否安全、是否能直接执行。

## Q9：executionAgent 在整个运维 Agent 中的执行流程是怎样的？如何保证安全退出而不是无休止调用工具？

### 推荐短答

`executionAgent` 采用 **状态驱动的迭代模型**，通过工具链流水线（Normalize/Generate → Validate → Execute → ValidateResult/Rollback）逐步执行修复步骤。安全退出依赖 **工具层熔断**（如重复执行保护、计数器限制）和 **门控 Agent 决策**（`executionGateAgent`）双重机制，确保在遇到错误或达到限制时优雅降级并请求人工介入。

### 展开回答

#### 1. executionAgent 的执行流程（工具链流水线）

`executionAgent` 并非一次性执行所有步骤，而是基于 **状态（State）** 驱动的迭代模型。它通过 LLM 决定下一步该调用哪个工具，形成闭环流水线：

1.  **计划生成/规范化**
    *   首先尝试 `NormalizePlanTool`：如果上游 `RemediationProposal` 的 `command_hint` 完整，直接转化为 `ExecutionPlan`，跳过 LLM 调用以提升效率。
    *   如果不完整，则调用 `GeneratePlanTool`，利用 LLM 生成详细计划。这里实现了**模板降级机制**：如果 LLM 调用失败（如超时、输出格式错误），会捕获错误并调用 `generatePlanWithTemplate()`，基于用户意图（如“重启”、“查看日志”）生成预定义的执行步骤，保证流程不中断。
    *   生成后立即调用 `validateExecutionPlanStructure`，确保 Steps 非空且命令必填。

2.  **静态风险校验**
    *   调用 `ValidatePlanTool`，使用正则三级匹配分析计划风险：
        *   **绝对禁止**（如 `rm -rf /`、`mkfs`）→ `blocked=true`
        *   **高风险变更**（如 `kubectl delete`、`docker stop`）→ `requires_confirmation`
        *   **只读操作**（如 `kubectl get`、`cat`）→ 安全
    *   通过 `markExecutionPlanValidated(ctx, planID)` 标记状态，这是后续执行的**前置守卫**。

3.  **命令执行（含 Human-in-the-Loop）**
    *   调用 `ExecuteStepTool`，这里有**多重前置检查**：
        *   必须已生成计划（`hasPreparedExecutionPlan`）。
        *   必须已通过风险校验（`hasValidatedExecutionPlan`）。
        *   **重复执行保护**：如果同一步骤同命令已成功执行，直接返回缓存结果，避免重复操作。
        *   **熔断机制**：
            *   同一命令连续失败 ≥2 次 → 标记 `manual_required` 并停止。
            *   未验证前执行 ≥4 次 → 标记 `manual_required` 并停止。
    *   检测到变更类命令（如 `kubectl delete`）时，调用 `tool.Interrupt()` 挂起，等待前端审批。

4.  **结果验证与回滚**
    *   执行后调用 `ValidateResultTool` 验证结果是否符合 `expected_result`。
    *   若失败且为关键步骤（`Critical=true`），调用 `RollbackTool` 执行回滚。

#### 2. 安全退出机制（避免无休止调用）

我们通过 **工具层熔断** + **门控 Agent 决策** 双重机制保证安全退出：

*   **工具层熔断（微观）**：
    *   `ExecuteStepTool` 内置计数器（`maxSameStepCommandAttempts = 2`），同一命令连续失败超限后，标记 `manual_required` 并停止执行，返回错误结果。
    *   通过 `shouldSkipExecutionStep` 检查，避免重复执行已成功的步骤。
    *   `GeneratePlanTool` 和 `ValidatePlanTool` 的模板降级机制，确保即使 LLM 失败，流程也不会中断或死循环。

*   **门控 Agent 决策（宏观）**：
    *   `executionAgent` 并不决定循环何时结束，而是由 **`executionGateAgent`**（在 `Loop(opsAgent, executionAgent, gate)` 中）负责。
    *   `executionGateAgent` 接收 `executionAgent` 的输出（`ExecutionResult`），根据以下条件决定是否退出 Loop：
        1.  **成功**：所有步骤执行完毕且成功。
        2.  **人工升级**：`executionAgent` 返回 `manual_required`（如熔断触发）。
        3.  **重复问题熔断**：同一问题重试超过 `maxRepeatedIssueRetries = 3` 次，强制退出循环。
        4.  **无更多步骤**：计划已执行完。
    *   一旦退出 Loop，流程进入 `strategyAgent` 生成报告，彻底结束运维工作流。

*   **双层熔断设计**：
    *   **内层**：execution 工具状态（`executionToolState`），负责 execution 内部重复失败止损。对 `generate_plan / validate_plan / execute_step / validate_result` 的重复失败做同类 Key 归一化（去掉时间戳、步骤号等噪声），达到阈值后直接在 execution 内部标记 `manual_required`。
    *   **外层**：`executionGateAgent`，负责 Loop 级重复重规划止损。
    *   这种双层设计确保既能控制 execution 内部空转，也能控制外层 Loop 的反复重规划。

#### 3. executionToolState 的存储机制

**关键更正**：`executionToolState` 不是保存在单独的 Redis Key 中，而是保存在 **SessionValues** 里，通过 Eino ADK 的上下文机制管理。

*   **存储位置**：`adk.GetSessionValue(ctx, "_execution_tool_state_v1")`
*   **生命周期**：随 Runner 的 Session 生命周期存在，中断恢复时通过 CheckPoint 序列化保存
*   **数据内容**：
    *   `planPrepared`：计划是否已准备
    *   `stepAttemptCount`：步骤执行次数
    *   `stepCommandAttemptCount`：同一命令尝试次数
    *   `repeatedFailureCount`：重复失败计数
    *   `stopExecution`：停止执行标志
*   **序列化机制**：通过 `GobEncode()` 和 `GobDecode()` 自定义序列化，确保中断恢复后状态不丢失

### 代码落点

*   `internal/agent/execution/agent.go`：executionAgent 核心逻辑
*   `internal/agent/execution/tools/generate_plan.go`：GeneratePlanTool（含模板降级）
*   `internal/agent/execution/tools/validate_plan.go`：ValidatePlanTool（静态风险分析）
*   `internal/agent/execution/tools/execute_step.go`：ExecuteStepTool（多重前置检查、熔断机制）
*   `internal/agent/execution/tools/tool_call_state.go`：executionToolState 状态管理（SessionValues）

*   `internal/agent/ops/incident_workflow.go`：Loop 编排

### 面试时可以这样总结

> `executionAgent` 采用状态驱动的迭代模型，通过工具链流水线逐步执行修复步骤。安全退出依赖双层熔断机制：工具层通过计数器和前置检查避免内部死循环，门控 Agent 通过重复问题计数和状态判断控制外层 Loop 收敛。
> 
> **关键点**：`executionToolState` 保存在 **SessionValues** 中，通过 Eino ADK 的上下文机制管理，中断恢复时通过 CheckPoint 序列化保存，确保状态不丢失。这种设计确保系统在遇到错误或达到限制时，能优雅降级并请求人工介入，而不是陷入无休止调用。

## Q10：GeneratePlanTool / ValidatePlanTool / ExecuteStepTool / ValidateResultTool 的运行流程是怎样的？

### 推荐短答

这四个工具构成 executionAgent 的核心流水线：GeneratePlanTool 生成计划（含 LLM 失败时的模板降级），ValidatePlanTool 静态风险分析，ExecuteStepTool 执行命令（含多重前置检查和熔断），ValidateResultTool 验证结果。整个流程基于状态驱动，确保每一步都有明确的前置条件和退出机制。

### 展开回答

#### 1. GeneratePlanTool（计划生成）

*   **输入**：上游 `RemediationProposal` 或用户意图/上下文。
*   **流程**：
    1.  解析输入，构建意图和上下文摘要。
    2.  尝试调用 LLM 生成 `ExecutionPlan`（包含步骤、命令、参数、预期结果、回滚命令等）。
    3.  如果 LLM 失败（超时、输出格式错误），捕获错误并调用 `generatePlanWithTemplate()`，基于用户意图生成预定义步骤（如“重启”→`systemctl restart`）。
    4.  验证计划结构（非空、命令必填），评估风险等级（low/medium/high）。
    5.  标记 `hasPreparedPlan`，保存计划到 ctx。
*   **安全机制**：模板降级确保即使 LLM 失败，流程也不会中断。

#### 2. ValidatePlanTool（风险校验）

*   **输入**：`ExecutionPlan`（可从 ctx 复用或直接传入）。
*   **流程**：
    1.  渲染每个步骤的完整命令文本。
    2.  使用正则三级匹配：
        *   **绝对禁止**（`rm -rf /`、`mkfs`、`dd if=`、`shutdown`、`fork bomb`）→ `blocked=true`
        *   **高风险变更**（`kubectl delete/drain/scale`、`docker stop/rm`、`systemctl stop/restart`）→ `requires_confirmation`
        *   **只读操作**（`kubectl get/describe/logs`、`cat/ls/ps`）→ 安全
    3.  计算风险分数，输出 `PlanValidationResult`（`valid`、`blocked`、`risk_level`）。
    4.  如果通过校验，标记 `hasValidatedPlan`。
*   **安全机制**：静态分析确定性判断，不依赖 LLM，确保风险识别的可靠性。

#### 3. ExecuteStepTool（命令执行）

*   **输入**：`step_id`（可省略，自动从计划中获取下一步）。
*   **流程**：
    1.  **前置检查**：
        *   必须已生成计划（`hasPreparedExecutionPlan`）。
        *   必须已通过风险校验（`hasValidatedExecutionPlan`）。
        *   检查是否跳过（`shouldSkipExecutionStep`）：如同一步骤同命令已成功，直接返回缓存结果。
        *   检查熔断：同一命令连续失败 ≥2 次，或未验证前执行 ≥4 次 → 标记 `manual_required` 并停止。
    2.  **白名单校验**：命令必须在 18 个允许命令列表中（bash/kubectl/systemctl/docker/echo/cat/ls/ps/curl/wget/ping/netstat/ss/top/df/du/free/uptime）。
    3.  **安全校验**（脚本模式）：禁止 `rm -rf /`、`mkfs`、`dd if=`、`shutdown/reboot`、`fork bomb`。
    4.  **变更类命令审批**：检测到变更操作（如 `kubectl delete`）时，调用 `tool.Interrupt()` 挂起，等待前端审批。
    5.  **实际执行**：bash 模式用 `exec.Command("bash", "-lc", script)`，直接模式用 `exec.Command(command, args...)`，带超时控制。
    6.  **输出裁剪**：`sanitizeExecutionOutput`（最多 8000 rune，头尾各半）。
    7.  保存结果到 ctx，返回 JSON 格式的 `ExecutionResult`。
*   **安全机制**：多重前置检查 + 熔断 + 人工审批，确保执行过程可控。

#### 4. ValidateResultTool（结果验证）

*   **输入**：`ExecutionResult`（从 ctx 获取或传入）。
*   **流程**：
    1.  对比实际输出与 `expected_result`（计划中定义）。
    2.  更新步骤验证状态到 ctx。
    3.  如果验证失败且为关键步骤，可触发回滚（但回滚通常由 `RollbackTool` 单独处理）。
*   **安全机制**：确保执行结果符合预期，否则标记失败并可能触发回滚。

### 整体运行流程示例

```
用户输入故障描述
    ↓
OpsAgent 生成 RemediationProposal
    ↓
executionAgent 启动：
    1. NormalizePlanTool（如果 Proposal 完整）→ 生成 ExecutionPlan
    2. GeneratePlanTool（如果不完整）→ LLM 生成或模板降级
    3. ValidatePlanTool → 静态风险分析
    4. ExecuteStepTool → 执行第一步（含审批/熔断）
    5. ValidateResultTool → 验证结果
    6. 重复 4-5 直到所有步骤完成
    ↓
executionGateAgent 判断：成功/人工升级/重复问题熔断/无更多步骤
    ↓
退出 Loop，进入 strategyAgent 生成报告
```

### 代码落点

*   `internal/agent/execution/tools/generate_plan.go`
*   `internal/agent/execution/tools/validate_plan.go`
*   `internal/agent/execution/tools/execute_step.go`
*   `internal/agent/execution/tools/validate_result.go`（假设存在）

### 面试时可以这样总结

> 这四个工具构成 executionAgent 的核心流水线：GeneratePlanTool 负责生成计划（含 LLM 失败时的模板降级），ValidatePlanTool 负责静态风险分析，ExecuteStepTool 负责执行命令（含多重前置检查和熔断），ValidateResultTool 负责验证结果。整个流程基于状态驱动，每一步都有明确的前置条件和退出机制，确保执行过程安全可控。

## Q11：executionGateAgent 的熔断是不是覆盖了所有重复失败？

### 推荐短答

不是。`executionGateAgent` 只负责 **外层工作流级别** 的重复问题收敛，前提是 `executionAgent` 这一轮已经结束并输出了可解析信号；它并不会直接感知 `executionAgent` 内部每一次工具调用失败。`generate_plan / validate_plan / execute_step / validate_result` 这类内部重复失败，实际是由 `executionToolState` 在 `executionAgent` 内部先做累计和止损。

### 展开回答

- `executionGateAgent` 处理的是 **Loop(opsAgent, executionAgent, gate)** 这一层的重复问题，它看到的不是原始 tool call，而是 `executionAgent` 一轮结束后产出的结构化结果或停止信号。
- 当前它主要收敛这些外层场景：
  - `step_validation_replan`
  - `execution_actionable_issue`
  - `execution_empty_steps`
  - `execution_replan`
- 换句话说，只有当 `executionAgent` 已经完成一轮内部调用，并把状态整理成“需要重规划 / 仍有未闭环问题 / 空步骤 / 失败原因”等信号后，Gate 才能对“同类问题是否反复出现”做计数。
- 但真实故障处理中，问题经常会卡在 `executionAgent` 内部，例如：
  - `generate_plan` 参数错误、计划结构校验反复失败
  - `validate_plan` 反复解析失败
  - `execute_step` 同一命令反复失败
  - `validate_result` 对同类失败连续校验不通过
- 这类问题如果只依赖 `executionGateAgent`，确实会在到达 Gate 之前就在 execution 内部空转。
- 所以当前实现是 **双层止损**：
  - **内层止损**：`executionToolState` 统一累计 `generate_plan / validate_plan / execute_step / validate_result` 的重复失败，对失败原因做 Key 归一化，去掉时间戳、步骤号、`plan_id`、`tooluse id` 等噪声；达到阈值后直接在 execution 内部标记 `manual_required`
  - **外层止损**：`executionGateAgent` 只负责外层 Loop 的重复重规划和重复问题收敛
- 两层之间的关系是：
  - 内层先防止 `executionAgent` 自己在工具调用阶段死循环
  - 外层再防止整个 `ops -> execution -> gate` 循环反复重规划不收敛

### 面试时可以这样总结

> `executionGateAgent` 不是“所有重复失败”的统一入口，它只负责外层工作流收敛。  
> `executionAgent` 内部的工具级重复失败，要靠 `executionToolState` 先做累计和止损。  
> 所以这套设计的关键不是单点熔断，而是“内层工具止损 + 外层工作流收敛”两层配合。

## Q12：运维 Agent 间只传递上一个 Agent 的输出对吗？如果一个 Agent 不断调用工具导致超过窗口怎么办？

### 推荐短答

运维 Agent 间通过 **结构化的 `IncidentState`（图状态）** 共享数据，而不是简单的“只传递上一个 Agent 的输出”。`IncidentState` 与用户会话记忆物理隔离，防止工具输出污染聊天上下文。针对工具调用超过窗口的问题，通过 **Token 预算控制 + 轮次压缩 + Thinking Block 过滤** 三层机制防护。

### 展开回答

#### 1. Agent 间的数据传递方式

**不是简单的“只传递上一个 Agent 的输出”，而是通过结构化的 `IncidentState` 进行共享：**

*   **数据载体**：`IncidentState`（定义在 `internal/agent/ops/state.go`），包含以下字段：
    *   `Observation`（故障上下文）
    *   `RootCause`（根因结论）
    *   `ProposalList`（修复提案列表）
    *   `ExecutionResults`（执行结果）
    *   `GateStatus`（门控状态）

*   **流转路径**：
    1.  `ObserverAgent` 初始化并更新 `Observation`
    2.  `RCA Agent` 读取 `Observation`，更新 `RootCause`
    3.  `OpsAgent` 读取 `RootCause`，更新 `ProposalList`
    4.  `ExecutionAgent` 读取 `ProposalList`，更新 `ExecutionResults`
    5.  `ExecutionGateAgent` 读取 `ExecutionResults`，更新 `GateStatus`
    6.  `StrategyAgent` 和 `FinalReportAgent` 读取完整的 `IncidentState` 生成报告

*   **物理隔离**：
    *   `IncidentState` 存储在 **Eino ADK 的图状态（Graph State）** 中，通过 Redis 持久化。
    *   用户对话历史存储在 **SessionMemory**（Redis 的 `aiagent:ctx:{id}:turns`）。
    *   **关键点**：工具调用日志（如 `kubectl get pods` 的输出）不会直接存入用户对话历史，而是存入 `IncidentState`，只有最终结论或需要用户确认的步骤才会通过 SSE 事件传给前端。

#### 2. 针对“不断调用工具导致超过窗口”的处理机制

如果一个 Agent 不断调用工具（如 `executionAgent` 连续执行 10 次 `execute_step`），我们通过三层机制防止上下文超过窗口：

**第一层：Token 预算硬控制（请求前裁剪）**
*   **实现位置**：`utility/mem/mem.go` -> `GetMessagesForRequest()`
*   **计算公式**：
    ```go
    turnsBudget = MaxInputTokens(96k)
                - ReserveOutputTokens(8k)
                - ReserveToolsTokens(20k)
                - UserTokens
                - SafetyTokens(2k)
                - SysTokens
    ```
*   **裁剪动作**：如果当前上下文超过 `turnsBudget`，使用 Redis Lua 脚本按 **Turn 粒度** 从最旧开始原子裁剪，直到总量 ≤ Budget。

**第二层：轮次压缩（Long-term Memory）**
*   **触发条件**：对话轮次超过 40 轮。
*   **压缩策略**：
    1.  保留最近 **20 轮** 完整对话。
    2.  将更早的轮次压缩成 1 段摘要（最大 1200 rune）。
    3.  压缩后的上下文 = `1 段摘要 + 20 轮完整`，大幅减少 Token 消耗。

**第三层：Thinking Block 过滤**
*   **实现**：`KeepReasoningInContext = false`
*   **动作**：过滤掉模型输出中的 `type: thinking` Block（长推理过程）。
*   **目的**：只保留最终的结构化输出，节省 Token 空间。

**针对工具调用的特殊处理：**
*   **输出截断**：`ExecuteStepTool` 将命令输出截断为最多 8000 rune（头尾各半），防止单次工具输出过大。
*   **状态传递**：Agent 间传递的是 `ExecutionResult` 结构体（包含 success、exit_code 等），而不是完整的 stdout/stderr 文本。

### 代码落点

| 机制 | 文件路径 | 核心函数 |
|------|----------|----------|
| IncidentState 定义 | `internal/agent/ops/state.go` | `type IncidentState struct {...}` |
| 工作流编排 | `internal/agent/ops/incident_workflow.go` | `Sequential(..., Loop(...))` |
| Token 预算计算 | `utility/mem/mem.go` | `GetMessagesForRequest()` |
| Lua 原子裁剪 | `utility/mem/mem.go` | `luaTrimTurnsToBudget()` |
| 轮次压缩 | `internal/context/session_memory.go` | `compactMessages()` |
| 工具输出截断 | `internal/agent/execution/tools/execute_step.go` | `sanitizeExecutionOutput()` |

### 面试时可以这样总结

> 运维 Agent 间通过 **结构化的 `IncidentState`** 共享数据，而不是简单的“输出传递”。`IncidentState` 与用户会话记忆物理隔离，防止工具输出污染聊天上下文。针对工具调用超过窗口的问题，我们通过 **Token 预算控制 + 轮次压缩 + Thinking Block 过滤** 三层机制防护，确保上下文始终在限制范围内。

## Q13：图状态（Graph State）是 Eino 框架提供的还是我自己设计的？

### 推荐短答

**图状态机制是 Eino ADK 框架原生提供的，但我设计了 `IncidentState` 这个具体的数据结构。** 我利用 Eino 的图状态机制存储和流转 `IncidentState`，实现 Agent 间解耦和状态持久化。

### 展开回答

#### 1. Eino 框架提供的能力

*   Eino ADK 的 `Runner` 在执行过程中维护一个全局的 **State（状态）**。
*   这个 State 可以在 Graph 的所有节点（Agent）之间共享和读写。
*   State 会随着 CheckPoint 一起序列化存储（如存入 Redis），支持中断恢复。
*   框架提供接口如 `WithState()`、`GetState()` 来绑定和访问状态。

#### 2. 我的设计：IncidentState

*   我定义了 `IncidentState` 结构体，专门用于存储故障处置的中间数据：
    ```go
    type IncidentState struct {
        Observation     string                `json:"observation"`
        RootCause       string                `json:"root_cause"`
        ProposalList    []RemediationProposal `json:"proposal_list"`
        ExecutionResults []ExecutionResult    `json:"execution_results"`
        GateStatus      string                `json:"gate_status"`
        // ... 其他字段
    }
    ```
*   我将 `IncidentState` 绑定到 Eino 的图状态机制中，利用框架提供的持久化能力。

#### 3. 为什么这样设计

*   **解耦**：通过结构化字段传递数据，避免 Agent 之间依赖自然语言的隐式约定。
*   **隔离**：与用户对话历史（SessionMemory）物理隔离，防止工具输出污染聊天上下文。
*   **可追溯**：每个阶段的输出都结构化存储，便于调试和复盘。

#### 4. 潜在追问与回答

**追问：Eino 的图状态是如何序列化和持久化的？**
> **回答：** Eino 的图状态通过 `CheckPointStore` 接口实现持久化。在我的项目中，我实现了 `RedisCheckPointStore`，将状态序列化（使用 `gob` 编码）后存入 Redis。中断时，Eino 自动将图状态（包含 `IncidentState`）序列化并存入 Redis（Key: `oncall:checkpoint:{id}`）；恢复时，从 Redis 读取 CheckPoint，反序列化后恢复图状态。

**追问：如果 Eino 框架升级，图状态机制变了，你的代码会受影响吗？**
> **回答：** 会有一定影响，但影响范围可控。受影响部分是与 Eino 框架直接交互的代码（如 `WithState`、`GetState`）；不受影响部分是 `IncidentState` 的结构定义和业务逻辑。通过封装 Eino 的状态访问接口，将框架依赖隔离在少数几个函数中，如果框架升级，只需修改封装层。

### 代码落点

| 功能 | 文件路径 | 说明 |
|------|----------|------|
| IncidentState 定义 | `internal/agent/ops/state.go` | 我设计的结构体 |
| Runner 初始化（绑定 State） | `internal/bootstrap/app.go` 或 `internal/controller/chat/chat_v1.go` | 利用 Eino 的 `WithState` |
| Agent 访问 State | `internal/agent/ops/ops_agent.go` 等 | 通过 Eino 上下文读取 State |
| CheckPoint 持久化 | `internal/context/checkpoint_store.go` | RedisCheckPointStore 实现 |

### 面试时可以这样总结

> **图状态机制是 Eino ADK 框架原生提供的，但我设计了 `IncidentState` 这个具体的数据结构。** 我利用 Eino 的图状态机制存储和流转 `IncidentState`，实现 Agent 间解耦、状态持久化和上下文隔离。这种设计既利用了框架能力，又保证了业务逻辑的灵活性和可维护性。

## Q14：SSE 是什么？为什么在项目中选择 SSE 而不是 WebSocket？

### 推荐短答

SSE (Server-Sent Events) 是 HTML5 标准中的单向通信技术，允许服务器向客户端实时推送数据。项目选择 SSE 是因为只需服务器向客户端推送流式数据（AI 生成内容、工具调用步骤、中断事件），无需双向通信，且实现简单、浏览器原生支持。

### 展开回答

#### 1. SSE 定义

- **SSE (Server-Sent Events)** 是基于 HTTP 长连接的单向通信技术
- **协议格式**：文本格式，每条消息以 `data: ` 开头，以 `\n\n` 结束
- **浏览器支持**：原生支持，使用 `EventSource` API

#### 2. SSE vs WebSocket 对比

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 单向（服务器 → 客户端） | 双向 |
| 协议 | HTTP/1.1+ | 独立协议 |
| 实现复杂度 | 简单（文本格式） | 较复杂（二进制/帧） |
| 适用场景 | 实时通知、日志流、AI 流式输出 | 即时通讯、游戏、协同编辑 |

#### 3. 为什么选择 SSE？

1. **单向通信足够**：项目只需服务器向客户端推送 AI 生成内容、工具调用步骤、中断事件，无需客户端向服务器发送大量数据
2. **实现简单**：SSE 是浏览器原生支持的标准，前端只需 `EventSource` API，后端只需设置响应头并按格式写入数据
3. **自动重连**：SSE 内置自动重连机制，网络中断后会自动恢复连接
4. **浏览器连接限制**：浏览器对同域 WebSocket 连接数无限制，但对 SSE 有 6 个连接限制。我们的项目每个会话只需 1 个 SSE 连接，完全够用

### 代码落点

- `internal/controller/chat/chat_v1.go` - SSE 初始化和数据写入
- `Front_page/src/services/api.ts` - 前端 SSE 消费逻辑

## Q15：SSE 在 OnCall 项目中是如何使用的？

### 推荐短答

OnCall 项目使用 SSE 实现流式通信，主要场景包括对话流式输出、运维工作流流式输出和中断恢复。后端通过 GoFrame 设置 SSE 响应头并按格式写入数据，前端使用 fetch API 消费 SSE 流。

### 展开回答

#### 1. SSE 端点

| 端点 | 作用 | 参数 |
|------|------|------|
| `/api/v1/chat_stream` | 对话流式输出 | `id` (sessionID), `question` |
| `/api/v1/chat_resume_stream` | 对话中断恢复 | `checkpoint_id`, `interrupt_ids`, `approved`, `resolved`, `comment` |
| `/api/v1/ai_ops_stream` | 运维工作流流式输出 | 无参数（固定诊断） |
| `/api/v1/ai_ops_resume_stream` | 运维工作流中断恢复 | `checkpoint_id`, `interrupt_ids`, `approved`, `resolved`, `comment` |

#### 2. 事件类型

OnCall 项目定义了 5 种 SSE 事件类型：

| 事件类型 | 格式 | 用途 |
|----------|------|------|
| `content` | `{"type":"content","content":"..."}` | AI 生成的文本内容 |
| `step` | `{"type":"step","step":N,"content":"..."}` | 工具调用步骤进度 |
| `interrupt` | `{"type":"interrupt","checkpoint_id":"...","message":"..."}` | 中断等待人工审批 |
| `error` | `{"type":"error","content":"..."}` | 错误信息 |
| `done` | `{"type":"done"}` 或 `[DONE]` | 流结束信号 |

#### 3. 后端实现

```go
// SSE 初始化
func setupSSE(ctx context.Context) (*ghttp.Request, error) {
    r := g.RequestFromCtx(ctx)
    r.Response.Header().Set("Content-Type", "text/event-stream")
    r.Response.Header().Set("Cache-Control", "no-cache")
    r.Response.Header().Set("Connection", "keep-alive")
    r.Response.Header().Set("X-Accel-Buffering", "no")
    r.Response.WriteHeader(200)
    r.Response.Flush()
    return r, nil
}

// 数据写入
func writeSSEData(r *ghttp.Request, data string) {
    lines := strings.Split(data, "\n")
    for _, line := range lines {
        r.Response.Write(fmt.Sprintf("data: %s\n", line))
    }
    r.Response.Write("\n")
    r.Response.Flush()
}
```

#### 4. 前端实现

```typescript
// SSE 消费逻辑
async function streamRequest(url: string, body: any, options: StreamOptions) {
    const response = await fetch(url, { method: 'POST', body: JSON.stringify(body) });
    const reader = response.body?.getReader();
    const decoder = new TextDecoder();
    let buffer = '';
    
    while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        
        buffer += decoder.decode(value, { stream: true });
        const parts = buffer.split('\n\n');  // SSE 事件分隔符
        buffer = parts.pop() || '';
        
        for (const part of parts) {
            const dataContent = part.split('\n')
                .filter(l => l.startsWith('data: '))
                .map(l => l.slice(6))
                .join('\n')
                .trim();
            
            // 解析 JSON 事件
            try {
                const json = JSON.parse(dataContent);
                switch(json.type) {
                    case 'content':   onContent(json.content); break;
                    case 'step':      onStep?.(json); break;
                    case 'interrupt': onInterrupt?.(mapInterruptData(json)); break;
                    case 'done':      onDone?.(); return;
                    case 'error':     onError?.(json.content); return;
                }
            } catch (e) {
                onContent(dataContent);
            }
        }
    }
}
```

## Q16：SSE 如何处理中断和恢复？

### 推荐短答

中断时，后端通过 SSE 推送 `interrupt` 事件，前端渲染审批卡片；恢复时，用户在前端提交决策，后端通过 `runner.ResumeWithParams()` 恢复执行，建立新的 SSE 连接继续推送内容。

### 展开回答

#### 1. 中断流程

1. 当 `ExecuteStepTool` 检测到高风险命令时，调用 `tool.Interrupt()` 挂起工作流
2. 后端通过 SSE 推送 `interrupt` 事件，包含：
   - `checkpoint_id`：标识中断点
   - `message`：中断信息（如待执行命令）
   - `interrupt_contexts`：中断上下文列表
3. 前端 `InterruptCard` 组件渲染审批卡片，展示待执行命令和作用说明

#### 2. 恢复流程

1. 用户在前端点击审批按钮（准许执行/拒绝/标记已解决）
2. 前端 POST 到恢复接口（`/chat_resume_stream` 或 `/ai_ops_resume_stream`）
3. 后端 `runner.ResumeWithParams()` 恢复执行：
   - 传入 `checkpoint_id` 标识恢复点
   - 传入 `interrupt_ids` 标识具体中断上下文
   - 传入 `approved`、`resolved`、`comment` 等用户决策
4. 工具内 `tool.GetResumeContext()` 取回用户决策
5. 建立新的 SSE 连接，继续推送后续内容

#### 3. 关键点

- `checkpoint_id` 用于标识中断点，支持任意位置恢复
- `interrupt_ids` 用于标识具体的中断上下文
- 恢复后，SSE 继续推送 `content`、`step`、`interrupt` 等事件

- `internal/agent/execution/tools/execute_step.go` - `tool.Interrupt()` 调用
- `internal/controller/chat/chat_v1.go` - `ChatResumeStream()` 和 `AIOpsResumeStream()`
- `Front_page/src/components/InterruptCard.tsx` - 中断审批和恢复逻辑
