# OnCall 项目介绍

## 1. 项目定位

`OnCall` 是一个面向 Kubernetes / 值班运维场景的多 Agent 智能运维系统，不是普通聊天机器人。

它要解决的核心问题是：传统故障处理依赖人工在 `kubectl`、Prometheus、Elasticsearch、历史案例和命令行之间来回切换，定位慢、执行风险高、经验难沉淀。  
我把这个流程工程化成了一个完整闭环：

`自然语言入口 -> 观测采集 -> 根因分析 -> 修复规划 -> 安全执行 -> 技术报告 -> 知识沉淀`

一句话介绍可以这样说：

> OnCall 是一个基于 `GoFrame + Eino ADK` 构建的多 Agent 运维自动化系统，能够围绕 Kubernetes 故障场景完成观测、RCA、执行审批、报告生成和知识沉淀的闭环。

---

## 2. 业务问题与设计目标

### 2.1 要解决的业务问题

- 运维排障依赖人工经验，定位链路长
- 故障信息分散在资源状态、监控指标、日志和历史案例中
- 大模型容易“会说不会做”，缺少受控执行能力
- 高风险命令不能直接执行，必须引入审批
- 故障处理是长流程，需要支持中断、恢复、回滚和转人工

### 2.2 设计目标

- 用自然语言统一运维入口
- 用多 Agent 分层降低复杂任务耦合
- 用工具调用获取事实，而不是让模型猜测
- 用审批、校验、熔断和回滚保证执行安全
- 用技术报告和知识库把一次处理沉淀成长期资产

## 4. 总体架构

### 4.1 后端总体分层

- **接入层**：GoFrame 提供 HTTP / SSE 接口
- **编排层**：Eino ADK 负责 Agent、Tool、Loop、Checkpoint 组合
- **能力层**：按职责拆分 `dialogue / knowledge / rca / ops / execution / strategy`
- **工具层**：封装 `k8s_monitor`、`metrics_collector`、`es_log_query`、`bash_execute_with_approval`
- **状态层**：分离会话记忆和执行 checkpoint
- **展示层**：React 前端消费 SSE，展示步骤流和审批卡片

### 4.2 核心调用链

```text
HTTP Request
  -> GoFrame Router
  -> Controller
  -> SessionMemory.BuildMessages()
  -> ADK Runner.Run() / Resume()
  -> Agent Workflow
  -> Tool Calls
  -> SSE: content / step / interrupt / done
```

## 5. 多 Agent 分工

### 5.1 Dialogue Agent

文件：`internal/agent/dialogue/agent.go:1`

职责：

- 理解用户意图
- 判断是普通问答、状态观测、故障处理还是知识检索
- 路由到合适工具或子流程

主要工具：

- `intent_analysis`
- `knowledge_retrieve`
- `ops_case_retrieve`
- `k8s_monitor`
- `metrics_collector`
- `web_search`
- `bash_execute_with_approval`

定位：对话入口和工具调度器，不负责深度运维编排。

### 5.2 Knowledge Agent

文件：`internal/agent/knowledge/agent.go:1`

职责：

- 上传知识文档
- 文档分片、索引、入库
- 返回结构化索引结果

定位：只负责知识沉淀，不混入运维执行逻辑。

### 5.3 RCA Agent

文件：`internal/agent/rca/agent.go:1`

职责：

- 基于 Kubernetes、Prometheus、时间窗口和依赖关系做根因分析
- 输出结构化 RCA 结果，而不是自由文本猜测

主要工具：

- `k8s_monitor`
- `metrics_collector`
- `time_query`
- `build_dependency_graph`
- `correlate_signals`
- `infer_root_cause`
- `analyze_impact`

输出重点：

- `root_cause`
- `target_node`
- `confidence`
- `evidence`
- `impact`
- `next_verification`

### 5.4 Ops Agent

文件：`internal/agent/ops/agent.go:1`

职责：

- 把 RCA 和观测结果转成结构化修复提案 `RemediationProposal`
- 如果现场状态和旧假设不一致，主动调整方案

定位：规划层，不直接执行资源变更。

### 5.5 Execution Agent

文件：`internal/agent/execution/agent.go:1`

职责：

- 把修复提案转换为可执行计划
- 逐步执行、逐步校验
- 必要时中断审批、回滚或转人工

核心工具链：

```text
normalize_plan
  -> generate_plan
  -> validate_plan
  -> execute_step
  -> validate_result
  -> rollback
```

定位：把模型输出约束成“可验证、可审批、可回滚”的执行流程。

### 5.6 Strategy Agent

文件：`internal/agent/strategy/agent.go:1`

职责：

- 复盘本次策略效果
- 总结有效/无效动作
- 更新知识库
- 清理低质量案例

定位：让系统不止处理一次故障，而是持续积累经验。

## 6. 运维主工作流

文件：`internal/agent/ops/incident_workflow.go:1`

项目没有直接使用框架内建的 plan-execute-replan Agent，而是自己用 `SequentialAgent + LoopAgent` 组合了一套更可控的 Incident Workflow：

```text
Observation Collector
  -> RCA Agent
  -> Loop(
       Ops Agent
       -> Execution Agent
       -> Execution Gate
     )
  -> Strategy Agent
  -> Final Reporter
```

### 6.1 这样设计的原因

- 需要在执行阶段插入人工审批
- 需要在每轮执行后做结果校验
- 需要在问题重复失败时熔断
- 需要把“规划”和“执行”彻底拆开

### 6.2 Execution Gate 的作用

文件：`internal/agent/ops/incident_nodes.go:1`

`Execution Gate` 不是多余节点，它负责决定本轮执行后流程该往哪走：

- 已成功修复：退出循环
- 计划失配：要求重规划
- 同类问题重复失败：停止自动化，转人工
- 仍需继续执行：进入下一轮

项目里还加入了重复问题重试上限，避免 Agent 陷入死循环。

## 7. 流式通信与中断恢复

### 7.1 为什么用 SSE

- 这个场景主要是服务端单向推流
- SSE 比 WebSocket 更轻量
- 实现和网关兼容性更简单
- 对 `content / step / interrupt / done` 这类事件流很合适

### 7.2 支持的流式接口

- `POST /api/v1/chat_stream`
- `POST /api/v1/chat_resume_stream`
- `POST /api/v1/ai_ops_stream`
- `POST /api/v1/ai_ops_resume_stream`

对应代码：

- 接口定义：`api/chat/v1/chat.go:1`
- 控制器实现：`internal/controller/chat/chat_v1.go:1`

### 7.3 中断恢复机制

高风险动作不会直接执行，而是走中断恢复流程：

1. 工具层发现需要审批
2. 当前 Runner 生成 interrupt 事件
3. 前端展示审批卡片
4. 用户选择批准 / 拒绝 / 已解决
5. 前端调用 resume 接口
6. 后端通过 checkpoint 从中断点继续执行

前端关键位置：

- SSE 请求封装：`Front_page/src/services/api.ts:1`
- 状态管理：`Front_page/src/store/useStore.ts:1`

## 8. 会话记忆与状态管理

### 8.1 为什么要把会话记忆和执行状态分开

这两个问题本质不同：

- **会话记忆**：解决上下文过长、token 成本过高的问题
- **执行状态**：解决中断、审批、断连后如何恢复流程的问题

如果混在一起，既会浪费 token，也不利于精确恢复。

### 8.2 Session Memory

文件：`internal/context/session_memory.go:1`

核心思路：

- 按轮次维护对话历史
- 超过预算后自动摘要和裁剪
- 避免把大段日志直接塞回模型上下文

这部分的价值不只是省 token，更重要的是避免模型被长日志干扰判断。

### 8.3 Checkpoint Store

文件：`internal/context/checkpoint_store.go:1`

核心思路：

- Runner 执行状态持久化到 Redis
- 支持按 `checkpoint_id` 恢复工作流
- 保证审批后能从正确节点继续，而不是整段重跑

## 9. 执行安全设计

这个项目最重要的工程价值之一，是把大模型能力放进一套确定性的执行约束里。

### 9.1 多层安全控制

- **命令白名单**：限制可执行命令类型
- **参数风险检查**：拦截危险片段和注入式参数
- **人工审批**：资源变更前必须确认
- **计划校验**：先 `validate_plan` 再 `execute_step`
- **结果校验**：执行后必须 `validate_result`
- **失败回滚**：必要时触发 `rollback`
- **重复问题熔断**：超过上限后停止自动化

### 9.2 为什么必须这样做

因为 LLM 可以辅助生成策略，但不能直接被信任为“最终执行器”。  
真正可上线的系统，必须把模型输出变成受控状态机。

## 10. 可观测性与事实驱动

项目对故障分析坚持“先事实、后建议”的原则。

### 10.1 事实来源

- Kubernetes 资源状态
- Prometheus 指标
- Elasticsearch 日志
- 历史故障案例
- 当前时间窗口与事件时间线

相关工具位置：

- `internal/agent/ops/tools/k8s_monitor.go:1`
- `internal/agent/ops/tools/metrics_collector.go:1`
- `internal/agent/ops/tools/es_log_query.go:1`

### 10.2 设计原则

- 先采集观测，再做根因推断
- 低证据时输出低置信度
- 缺少关键上下文时优先追问
- 不允许“编造根因”

## 11. 四个最值得讲的亮点

### 亮点一：多 Agent 分层明确

- 对话、RCA、规划、执行、复盘、知识沉淀彻底拆开
- 每个 Agent 只做一类决策
- 降低了 prompt 污染和职责耦合

面试中可强调：这体现的是系统分层和边界设计能力，而不是简单接模型 API。

### 亮点二：长流程状态管理完整

- 既有会话记忆裁剪，也有 checkpoint 恢复
- 既考虑 token 成本，也考虑流程可恢复性
- 故障处理更像工作流，而不是一次请求响应

面试中可强调：这是典型的后端状态管理问题，不只是 AI 问答问题。

### 亮点三：执行安全和稳定性做得比较扎实

- 高风险操作审批
- 执行结果校验
- 回滚与人工接管
- 重复问题熔断

面试中可强调：我关注的不是“能不能跑”，而是“能不能安全上线”。

### 亮点四：形成了运维闭环

- 不只是看状态
- 不只是做分析
- 而是把观测、分析、执行、报告、知识沉淀串成一条链路

面试中可强调：这不是 demo，而是一个具备长期演进价值的运维系统。

## 13. 面试官高频追问与回答思路

### 13.1 为什么要拆成多个 Agent

答法：

- 对话理解、根因分析、修复规划、执行控制是不同问题
- 放进一个 Agent 会导致上下文混乱、输出不稳定
- 分层后更容易控制权限、调试问题和扩展能力

### 13.2 为什么要做 checkpoint

答法：

- 故障处理是长流程，不是一次函数调用
- 中间会出现审批、断连、人工确认和失败重试
- 没有 checkpoint 就无法保证流程连续性

### 13.3 为什么执行层要单独拆出来

答法：

- 执行是高风险动作，不能直接信任上游推理结果
- 必须在执行前做计划规范化和风险校验
- 执行后还要做结果验证和回滚决策

### 13.4 为什么要做会话裁剪

答法：

- 运维日志和指标输出非常长
- 全部塞进上下文会增加成本，也会让模型失焦
- 所以要把对话记忆和故障状态分开管理

### 13.5 这个项目最像后端工程的部分是什么

答法：

- 流式接口设计
- 中断恢复
- 状态持久化
- 权限与审批控制
- 重试与熔断
- 工具链与外部系统集成

## 16. 总结

`OnCall` 的核心价值，不是“让模型帮我回答问题”，而是把故障处理这个高风险、长链路、强依赖事实的过程，抽象成一个可恢复、可审批、可校验、可沉淀的后端系统。

如果面试要突出后端能力，重点讲这四件事：

- 多 Agent 分层与职责边界
- SSE + checkpoint 的长流程状态管理
- 执行安全控制与熔断机制
- 运维闭环和知识沉淀能力
