# OnCall 多 Agent 运维系统 - 面试亮点

## 亮点一：可中断恢复的 Human-in-the-Loop 工作流

**技术深度：Eino ADK + CheckPointStore + Redis 持久化**

- **核心实现**：基于 ByteDance Eino ADK 的 `ResumableAgent`，在 `ExecuteStepTool` 中调用 `tool.Interrupt()` 主动挂起工作流，通过 Redis 持久化检查点（`oncall:checkpoint:{id}`, TTL 24h），支持任意中断点恢复。
- **关键技术**：
  - `tool.Interrupt(ctx, approvalInfo)`：在变更类命令执行前触发中断，等待用户审批
  - `tool.GetResumeContext()`：恢复时在工具内取回用户决策（approved/resolved/comment）
  - `runner.ResumeWithParams(ctx, checkpointID, params)`：从中断点继续执行完整工作流
- **前端联动**：`InterruptCard` 组件渲染审批卡片，收集用户决策后 POST 到 `/chat_resume_stream`，恢复 SSE 流式输出。
- **业务价值**：高风险命令强制人工审批，避免误操作；任意中断点可完整恢复，无需重跑整个流程。

## 亮点二：五阶段 Multi-Agent 分层工作流编排

**技术深度：Sequential + Loop 智能编排 + Graph State 物理隔离**

- **核心架构**：
```
Sequential(
  observerAgent,        // 阶段1: 观察 - 收集故障上下文
  rcaAgent,             // 阶段2: RCA - 7个工具根因分析
  Loop(                 // 阶段3: 循环修复
    opsAgent,           //   - 生成 RemediationProposal
    executionAgent,     //   - 6工具命令级执行
    executionGateAgent  //   - 门控: 熔断/升级/退出判断
  ),
  strategyAgent,        // 阶段4: 策略 - 经验沉淀
  finalReportAgent      // 阶段5: 收口 - Markdown报告+归档
)
```
- **关键技术**：
  - **双层熔断**：`executionGateAgent` 负责外层 `Loop` 的重复重规划熔断；`execution` 工具状态负责 `generate_plan / validate_plan / execute_step / validate_result` 的内部重复失败累计，避免问题在到达 Gate 之前就在 execution 内部死循环
  - **状态隔离**：图状态（`IncidentState`）存储诊断数据，会话记忆（`SessionMemory`）仅存用户可见对话，物理隔离避免污染
  - **工具链**：RCA 7工具（K8sMonitor/MetricsCollector/CorrelateSignals等）+ Execution 6工具（Normalize/Generate/Validate/Execute/Result/Rollback）
- **业务价值**：故障诊断到修复全流程自动化，分层解耦，职责单一，易于维护扩展。

## 亮点三：精准 Token 预算管理与原子裁剪

**技术深度：Redis + Lua 脚本 + 两层记忆设计**

- **核心实现**：
  - **两层设计**：`utility/mem/mem.go`（底层 Redis + Token 预算）+ `internal/context/session_memory.go`（上层 compact 逻辑）
  - **Token 预算公式**：
```
turnsBudget = MaxInputTokens(96k)
            - ReserveOutputTokens(8k)
            - ReserveToolsTokens(20k)
            - UserTokens - SafetyTokens(2k) - SysTokens
```
- **关键技术**：
  - **Lua 原子脚本**：`luaTrimTurnsToBudget` 在 Redis 侧原子执行历史裁剪，按 Turn 粒度从最旧开始弹出，避免多实例竞争
  - **摘要压缩**：超过 40 轮触发 LLM 摘要压缩，保留最近 20 轮 + 历史摘要（最大 1200 rune）
  - **Thinking Block 过滤**：`KeepReasoningInContext=false` 过滤 32k thinking block，避免推理过程占用历史空间
- **业务价值**：精确控制上下文成本，避免 Token 超限导致 API 报错；大日志/大对象不直接回灌模型。

## 亮点四：白名单+多层安全防护的命令执行

**技术深度：命令白名单 + 静态风险分析 + 运行时校验**

- **核心实现**：
  - **白名单机制**：`ExecuteStepTool` 内置 18 个允许命令（bash/kubectl/systemctl/docker/echo/cat/ls/ps/curl/wget/ping/netstat/ss/top/df/du/free/uptime）
  - **静态风险分析**：`ValidatePlanTool` 使用正则三级匹配：
    - 绝对禁止（rm -rf /、mkfs、dd if=、shutdown、fork bomb）→ blocked=true
    - 高风险变更（kubectl delete/drain/scale、docker stop/rm、systemctl stop）→ requires_confirmation
    - 只读操作（kubectl get/describe/logs、cat/ls/ps）→ 安全
  - **运行时校验**：
    - 脚本模式：逐段解析（|/&&/;/\n 分割），每段独立判断只读性
    - 超时控制：context.WithTimeout 防止长时间阻塞
    - 输出裁剪：sanitizeExecutionOutput（最多 8000 rune，头尾各半）
- **业务价值**：多层级防护确保高风险操作必须人工审批，防止误操作导致生产事故；只读操作自动执行提升效率。
